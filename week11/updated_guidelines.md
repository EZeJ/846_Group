### Problem B: Sales Revenue Aggregator with ETL Retry Deduplication

**Task Description:**  
Given an integer array of length n (up to 2×10⁵), implement a query processor that handles three interleaved operations: range add (add x to all elements in [l, r]), range set (set all elements in [l, r] to x), and range sum (return the sum of elements in [l, r]).

Your solution must efficiently handle up to 2×10⁵ operations in the worst case. Since updates and queries are fully interleaved, naively recomputing ranges will not scale — the implementation must support all three operations in sublinear time.

**Starter Code:**
```python
class RangeQueryProcessor:
    def __init__(self, nums):
        self.nums = nums[:]

    def add_range(self, left, right, value):
        for i in range(left, right + 1):
            self.nums[i] += value

    def set_range(self, left, right, value):
        for i in range(left, right + 1):
            self.nums[i] = value

    def range_sum(self, left, right):
        total = 0
        for i in range(left, right + 1):
            total += self.nums[i]
        return total


def process_queries(nums, operations):
    processor = RangeQueryProcessor(nums)
    results = []

    for op in operations:
        if op[0] == "add":
            _, l, r, x = op
            processor.add_range(l, r, x)
        elif op[0] == "set":
            _, l, r, x = op
            processor.set_range(l, r, x)
        elif op[0] == "sum":
            _, l, r = op
            results.append(processor.range_sum(l, r))

    return results
```

class RangeQueryProcessor:

        self.nums = nums[:]
        self.prefix = self._build_prefix()

    def _build_prefix(self):
        prefix = [0] * (len(self.nums) + 1)
        for i, value in enumerate(self.nums):
            prefix[i + 1] = prefix[i] + value
        return prefix

**Problems with produced output**
The output is cleaner and uses prefix sums for O(1) sum queries, but it still fails on worst-case inputs. Every "add" or "set" operation touches the entire range and rebuilds the full prefix array, making updates O(n) — with up to 2×10⁵ interleaved operations, this won't scale.

The deeper issue is that the guideline pushes for a visible optimization without forcing reasoning about update/query asymmetry or data structure selection. The result looks reasonable but still can't handle the actual worst-case workload.

### 3. New and Updated Guidelines that worked for Problem B

#### Guideline 3: Ask for worst-case performance over the full operation mix and require an appropriate data structure

For problems with multiple interleaved operation types, instructing the model to focus on large inputs is insufficient. This approach might enhance one operation while neglecting another, which becomes the real bottleneck. In such cases, a solution that speeds up queries but still performs updates in linear time will struggle at worst-case scale. Clearly requesting worst-case performance across the entire operation mix encourages the model to select a data structure that maintains efficiency for both updates and queries, instead of relying solely on local fixes that do not scale.

**Prompt and Context:**  
```text
The current implementation is too slow for worst-case inputs.
Assume n and the number of operations can both be up to 2 * 10^5.
The solution must remain correct for interleaved range updates and range sum queries.
Do not optimize only one operation type while leaving another at linear cost.
Choose a data structure that keeps both updates and queries efficient in the worst case.
Avoid rebuilding full arrays or rescanning full ranges after each update.
Prefer logarithmic-time operations if possible, and use lazy propagation if needed.
```

**Solution or Output generated by GitHub Copilot:**  

```python
class SegmentTree:
    def __init__(self, nums):
        self.n = len(nums)
        self.tree = [0] * (4 * self.n)
        self.lazy_add = [0] * (4 * self.n)
        self.lazy_set = [None] * (4 * self.n)
        self._build(1, 0, self.n - 1, nums)

    def _build(self, node, left, right, nums):
        if left == right:
            self.tree[node] = nums[left]
            return
        mid = (left + right) // 2
        self._build(node * 2, left, mid, nums)
        self._build(node * 2 + 1, mid + 1, right, nums)
        self.tree[node] = self.tree[node * 2] + self.tree[node * 2 + 1]

    def _apply_set(self, node, left, right, value):
        self.tree[node] = (right - left + 1) * value
        self.lazy_set[node] = value
        self.lazy_add[node] = 0

    def _apply_add(self, node, left, right, value):
        self.tree[node] += (right - left + 1) * value
        if self.lazy_set[node] is not None:
            self.lazy_set[node] += value
        else:
            self.lazy_add[node] += value

    def _push(self, node, left, right):
        if left == right:
            return

        mid = (left + right) // 2
        left_child = node * 2
        right_child = node * 2 + 1

        if self.lazy_set[node] is not None:
            self._apply_set(left_child, left, mid, self.lazy_set[node])
            self._apply_set(right_child, mid + 1, right, self.lazy_set[node])
            self.lazy_set[node] = None

        if self.lazy_add[node] != 0:
            self._apply_add(left_child, left, mid, self.lazy_add[node])
            self._apply_add(right_child, mid + 1, right, self.lazy_add[node])
            self.lazy_add[node] = 0

    def range_add(self, ql, qr, value):
        self._range_add(1, 0, self.n - 1, ql, qr, value)

    def _range_add(self, node, left, right, ql, qr, value):
        if ql > right or qr < left:
            return
        if ql <= left and right <= qr:
            self._apply_add(node, left, right, value)
            return

        self._push(node, left, right)
        mid = (left + right) // 2
        self._range_add(node * 2, left, mid, ql, qr, value)
        self._range_add(node * 2 + 1, mid + 1, right, ql, qr, value)
        self.tree[node] = self.tree[node * 2] + self.tree[node * 2 + 1]

    def range_set(self, ql, qr, value):
        self._range_set(1, 0, self.n - 1, ql, qr, value)

    def _range_set(self, node, left, right, ql, qr, value):
        if ql > right or qr < left:
            return
        if ql <= left and right <= qr:
            self._apply_set(node, left, right, value)
            return

        self._push(node, left, right)
        mid = (left + right) // 2
        self._range_set(node * 2, left, mid, ql, qr, value)
        self._range_set(node * 2 + 1, mid + 1, right, ql, qr, value)
        self.tree[node] = self.tree[node * 2] + self.tree[node * 2 + 1]

    def query_sum(self, ql, qr):
        return self._query_sum(1, 0, self.n - 1, ql, qr)

    def _query_sum(self, node, left, right, ql, qr):
        if ql > right or qr < left:
            return 0
        if ql <= left and right <= qr:
            return self.tree[node]

        self._push(node, left, right)
        mid = (left + right) // 2
        return (
            self._query_sum(node * 2, left, mid, ql, qr) +
            self._query_sum(node * 2 + 1, mid + 1, right, ql, qr)
        )


def process_queries(nums, operations):
    if not nums:
        return []

    seg = SegmentTree(nums)
    results = []

    for op in operations:
        if op[0] == "add":
            _, l, r, x = op
            seg.range_add(l, r, x)
        elif op[0] == "set":
            _, l, r, x = op
            seg.range_set(l, r, x)
        elif op[0] == "sum":
            _, l, r = op
            results.append(seg.query_sum(l, r))

    return results
```

**Why is it better?**

The updated guideline worked by forcing optimization across the entire workload rather than a single operation. Weak prompts tend to produce local improvements, such as prefix sums, caching, cleaner loops, that still leave updates or queries at linear cost.

The stronger guideline forces reasoning about the full worst-case mix of updates and queries, requiring a data structure where neither is O(n). That naturally pushes toward a segment tree with lazy propagation, which is the correct solution for this problem.
