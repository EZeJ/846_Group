            user_id, phone_number,
        )
        return {"success": False, "error": "invalid_phone"}

    profile.phone_number = phone_number
    profile.date_of_birth = date_of_birth
    profile.shipping_address = shipping_address

    saved = save_user_profile(profile)
    logger.info(
        "profile updated: user_id=%s, email=%s, phone_number=%s, "
        "date_of_birth=%s, shipping_address=%s, saved=%s",
        user_id, email, phone_number, date_of_birth, shipping_address, saved,
    )

        logger.error("db write failed: user_id=%s", user_id)

**Why this is not optimal:**

Guideline 7 says to log input parameters and key intermediate states so Copilot does exactly that — it logs `email`, `phone_number`, `date_of_birth` and `shipping_address` without any masking. The issue is that unlike `password` these field names don't look obviously sensitive to the model so it just logs them all. In real projects, logs usually get shipped to tools like Datadog or Splunk so this personal data ends up stored in plaintext somewhere outside the main database, which is a GDPR violation. The guideline simply doesn't mention anything about PII or what fields to exclude so Copilot has no way to know it's doing something wrong.

### 3. New and Updated Guidelines that worked for Problem A_1

### Guideline 7 (Updated): Provide Rich and Structured Context with a PII Exclusion Requirement

When asking Copilot to add logging always specify which fields are safe to log and which ones must never appear in logs. Put a clear PII exclusion list directly in the prompt and only allow safe identifiers like internal IDs, error labels and boolean outcomes.

**Prompt and Context:**  

```
I have /Users/sofiiatkach/Desktop/LLM/846_Group/week12/resources/updated_guideline_7/user_profile_service.py. Add logging to the update_user_profile function. Requirements:
- Include relevant runtime context to help debug failures and understand
  execution flow.
- Log user_id and outcome (success or error type) at every decision point.

Privacy constraints — the following fields contain PII and must NEVER appear
in any log statement:
- email
- phone_number
- date_of_birth
- shipping_address
- any field from the UserProfile dataclass other than user_id

Do not log the full profile object. Do not echo user-supplied values back
into warning or error messages.
```

**Solution or Output generated by GitHub Copilot:**  

```python
def update_user_profile(
    user_id: str,
    email: str,
    phone_number: str,
    date_of_birth: str,
    shipping_address: str,
) -> dict:
    logger.info("update_user_profile started: user_id=%s", user_id)

    profile = get_user_profile(user_id)

    if profile is None:
        logger.warning("update failed: user_not_found, user_id=%s", user_id)
        return {"success": False, "error": "user_not_found"}

    if not validate_email(email):
        logger.warning("update failed: invalid_email, user_id=%s", user_id)
        return {"success": False, "error": "invalid_email"}

    if not validate_phone(phone_number):
        logger.warning("update failed: invalid_phone, user_id=%s", user_id)
        return {"success": False, "error": "invalid_phone"}

    profile.email = email
    profile.phone_number = phone_number
    profile.date_of_birth = date_of_birth
    profile.shipping_address = shipping_address

    saved = save_user_profile(profile)

    if not saved:
        logger.error("db write failed: user_id=%s", user_id)
        return {"success": False, "error": "db_write_failed"}

    logger.info("update_user_profile succeeded: user_id=%s", user_id)
    return {"success": True, "error": None}
```
**Rationale**
The updated guideline adds an explicit PII exclusion list to the prompt, so Copilot knows exactly which fields to skip. Without it, the model just logs everything it sees.

### 3. New and Updated Guidelines that worked for Problem B

#### Guideline 6:  Enforce a Strict Logging Budget with Explicit Rules

**Description**

When prompting an LLM to add logging, define a strict and enforceable logging budget.
Limit logs to at most one INFO message per major stage, allow WARNING only for recoverable anomalies (e.g., dropped or invalid data), and use ERROR only for failures that prevent successful completion. Explicitly forbid per-record, per-loop, or helper-level logging unless specifically requested. Additionally, require that each log includes predefined, task-relevant fields (e.g., counts, identifiers, retry metadata).

**Reasoning**

LLMs tend to overproduce logs when given vague instructions such as “add helpful logging,” often resulting in verbose, repetitive, or low-signal outputs. While the original guideline constrains logging conceptually, making the constraints explicit and rule-based significantly improves adherence. A strict logging budget reduces noise, enforces consistent severity usage, and ensures logs remain actionable and comparable across runs. By explicitly defining both limits and required fields, the prompt shifts from descriptive guidance to enforceable structure, leading to higher-quality and more production-ready logging behavior.
 
**Example:**

**Bad Example:**
Add logging to this API retry workflow.


**Good Example:**

You are improving logging in a Python codebase.

Task:
Add logging to this candidate ranking workflow using a strict logging budget.

Strict logging budget:
- Emit at most 1 INFO log per major stage:
  1) pipeline start
  2) filtering summary
  3) deduplication summary
  4) final ranking outcome
- Emit WARNING only for recoverable anomalies:
  - invalid candidates dropped
  - duplicate candidates removed
  - retryable API failures
- Emit ERROR only for stage-level failures that prevent successful completion:
  - profile score fetch failed after all retries
- Do not add per-record, per-candidate, per-loop-iteration, or per-helper-call logs unless explicitly requested.
- Do not narrate obvious control flow.

Required fields:
- For filtering logs: before_count, after_count, dropped_count
- For deduplication logs: before_count, after_count, duplicate_count
- For retry warnings: email, attempt, max_retries, backoff_s
- For retry exhaustion errors: email, attempts_used
- For final outcome logs: requested_top_k, returned_count

Additional requirements:
- Use Python’s logging library only.
- Do not use print() for logging.
- Prefer structured, high-signal log messages with explicit fields.
- Keep DEBUG logs out unless absolutely necessary.
- Return the full updated code for all changed files.


**Prompt and Context:**  

Good example prompt for the G6.

**Solution or Output generated by GitHub Copilot:**  


The updated guideline introduced strict, enforceable rules, which changed the model’s behavior.

- It forbids per-record logging and pushes the model toward summaries:

~~~python
logger.info(
    "Filtering summary: before_count=%d after_count=%d dropped_count=%d",
    before_count, after_count, dropped_count
)
~~~

- It enforces stage-level logging instead of narration:

~~~python
logger.info(
    "Final ranking outcome: requested_top_k=%d returned_count=%d",
    top_k, len(result)
)
~~~

- It standardizes structured logging:

~~~python
logger.warning(
    "Retryable API failure: email=%s attempt=%d max_retries=%d backoff_s=%.2f",
    email, attempt, max_retries, backoff_s * attempt
)
~~~

- It makes severity mapping clearer:
  - INFO for summaries
  - WARNING for anomalies
  - ERROR for failures

Result: concise, structured, high-signal, production-ready logs.
