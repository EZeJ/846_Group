def update_user_profile(
    user_id: str,
    email: str,
    phone_number: str,
    date_of_birth: str,
    shipping_address: str,
) -> dict:
    logger.info(
        "update_user_profile called: user_id=%s, email=%s, phone_number=%s, "
        "date_of_birth=%s, shipping_address=%s",
        user_id, email, phone_number, date_of_birth, shipping_address,
    )

    profile = get_user_profile(user_id)
    logger.debug("fetched profile: %s", profile)

    if profile is None:
        logger.warning("update failed: user not found, user_id=%s", user_id)
        return {"success": False, "error": "user_not_found"}

    if not validate_email(email):
        logger.warning(
            "update failed: invalid email, user_id=%s, email=%s",
            user_id, email,
        )
        return {"success": False, "error": "invalid_email"}

    if not validate_phone(phone_number):
        logger.warning(
            "update failed: invalid phone, user_id=%s, phone_number=%s",
            user_id, phone_number,
        )
        return {"success": False, "error": "invalid_phone"}

    profile.email = email
    profile.phone_number = phone_number
    profile.date_of_birth = date_of_birth
    profile.shipping_address = shipping_address

    saved = save_user_profile(profile)
    logger.info(
        "profile updated: user_id=%s, email=%s, phone_number=%s, "
        "date_of_birth=%s, shipping_address=%s, saved=%s",
        user_id, email, phone_number, date_of_birth, shipping_address, saved,
    )

    if not saved:
        logger.error("db write failed: user_id=%s", user_id)
        return {"success": False, "error": "db_write_failed"}

    return {"success": True, "error": None}
```


**Why this is not optimal:**

Guideline 7 says to log input parameters and key intermediate states so Copilot does exactly that — it logs `email`, `phone_number`, `date_of_birth` and `shipping_address` without any masking. The issue is that unlike `password` these field names don't look obviously sensitive to the model so it just logs them all. In real projects, logs usually get shipped to tools like Datadog or Splunk so this personal data ends up stored in plaintext somewhere outside the main database, which is a GDPR violation. The guideline simply doesn't mention anything about PII or what fields to exclude so Copilot has no way to know it's doing something wrong.

### 3. New and Updated Guidelines that worked for Problem A_1

### Guideline 7 (Updated): Provide Rich and Structured Context with a PII Exclusion Requirement

When asking Copilot to add logging always specify which fields are safe to log and which ones must never appear in logs. Put a clear PII exclusion list directly in the prompt and only allow safe identifiers like internal IDs, error labels and boolean outcomes.

**Prompt and Context:**  

```
I have /Users/sofiiatkach/Desktop/LLM/846_Group/week12/resources/updated_guideline_7/user_profile_service.py. Add logging to the update_user_profile function. Requirements:
- Include relevant runtime context to help debug failures and understand
  execution flow.
- Log user_id and outcome (success or error type) at every decision point.

Privacy constraints — the following fields contain PII and must NEVER appear
in any log statement:
- email
- phone_number
- date_of_birth
- shipping_address
- any field from the UserProfile dataclass other than user_id

Do not log the full profile object. Do not echo user-supplied values back
into warning or error messages.
```

**Solution or Output generated by GitHub Copilot:**

        logger.warning("update failed: invalid_phone, user_id=%s", user_id)

**Rationale**
The updated guideline adds an explicit PII exclusion list to the prompt, so Copilot knows exactly which fields to skip. Without it, the model just logs everything it sees.

### 3. New and Updated Guidelines that worked for Problem B

#### Guideline 6:  Enforce a Strict Logging Budget with Explicit Rules

**Description**

When prompting an LLM to add logging, define a strict and enforceable logging budget.
Limit logs to at most one INFO message per major stage, allow WARNING only for recoverable anomalies (e.g., dropped or invalid data), and use ERROR only for failures that prevent successful completion. Explicitly forbid per-record, per-loop, or helper-level logging unless specifically requested. Additionally, require that each log includes predefined, task-relevant fields (e.g., counts, identifiers, retry metadata).

**Reasoning**

LLMs tend to overproduce logs when given vague instructions such as “add helpful logging,” often resulting in verbose, repetitive, or low-signal outputs. While the original guideline constrains logging conceptually, making the constraints explicit and rule-based significantly improves adherence. A strict logging budget reduces noise, enforces consistent severity usage, and ensures logs remain actionable and comparable across runs. By explicitly defining both limits and required fields, the prompt shifts from descriptive guidance to enforceable structure, leading to higher-quality and more production-ready logging behavior.
 
**Example:**

**Bad Example:**
Add logging to this API retry workflow.


**Good Example:**

You are improving logging in a Python codebase.

Task:
Add logging to this candidate ranking workflow using a strict logging budget.

Strict logging budget:
- Emit at most 1 INFO log per major stage:
  1) pipeline start
  2) filtering summary
  3) deduplication summary
  4) final ranking outcome
- Emit WARNING only for recoverable anomalies:
  - invalid candidates dropped
  - duplicate candidates removed
  - retryable API failures
- Emit ERROR only for stage-level failures that prevent successful completion:
  - profile score fetch failed after all retries
- Do not add per-record, per-candidate, per-loop-iteration, or per-helper-call logs unless explicitly requested.
- Do not narrate obvious control flow.

Required fields:
- For filtering logs: before_count, after_count, dropped_count
- For deduplication logs: before_count, after_count, duplicate_count
- For retry warnings: email, attempt, max_retries, backoff_s
- For retry exhaustion errors: email, attempts_used
- For final outcome logs: requested_top_k, returned_count

Additional requirements:
- Use Python’s logging library only.
- Do not use print() for logging.
- Prefer structured, high-signal log messages with explicit fields.
- Keep DEBUG logs out unless absolutely necessary.
- Return the full updated code for all changed files.


**Prompt and Context:**  

Good example prompt for the G6.

**Solution or Output generated by GitHub Copilot:**  


The updated guideline introduced strict, enforceable rules, which changed the model’s behavior.

- It forbids per-record logging and pushes the model toward summaries:

~~~python
logger.info(
    "Filtering summary: before_count=%d after_count=%d dropped_count=%d",
    before_count, after_count, dropped_count
)
~~~

- It enforces stage-level logging instead of narration:

~~~python
logger.info(
    "Final ranking outcome: requested_top_k=%d returned_count=%d",
    top_k, len(result)
)
~~~

- It standardizes structured logging:

~~~python
logger.warning(
    "Retryable API failure: email=%s attempt=%d max_retries=%d backoff_s=%.2f",
    email, attempt, max_retries, backoff_s * attempt
)
~~~

- It makes severity mapping clearer:
  - INFO for summaries
  - WARNING for anomalies
  - ERROR for failures

Result: concise, structured, high-signal, production-ready logs.

def label_join_outcomes(merged: pd.DataFrame) -> pd.DataFrame:
    before = len(merged)
    labeled = merged.copy()
    labeled["join_outcome"] = labeled["status"].fillna("missing_user")
    after = len(labeled)

    outcome_counts = labeled["join_outcome"].value_counts().to_dict()
    logging.info(
        "Labeled join outcomes: %d -> %d rows. Outcome counts: %s",
        before,
        after,
        outcome_counts,
    )
    return labeled
~~~

~~~python
def aggregate_country_revenue(merged: pd.DataFrame) -> pd.DataFrame:
    before = len(merged)
    report = (
        merged.groupby("country", as_index=False)["amount"]
        .sum()
        .rename(columns={"amount": "total_revenue"})
    )
    after = len(report)

    logging.info(
        "Aggregated revenue by country: %d input rows -> %d country groups.",
        before,
        after,
    )
    logging.info("Aggregated revenue by country:\n%s", report.to_string(index=False))
    return report
~~~

~~~python
def sort_report(report: pd.DataFrame) -> pd.DataFrame:
    before = len(report)
    sorted_report = report.sort_values(
        "total_revenue", ascending=False
    ).reset_index(drop=True)
    after = len(sorted_report)

    logging.info(
        "Sorted report by total_revenue descending: %d -> %d rows.",
        before,
        after,
    )
    return sorted_report
~~~

**Why this is not optimal:**

The original Guideline 1 says to log stages that change the data in a meaningful way, but that phrase is too vague. Copilot followed it literally and started logging transformations that are technically changes but not operationally useful. `label_join_outcomes()` only adds a column, yet the model still logs row counts and outcome labels. `sort_report()` only reorders rows, yet it still receives a before/after log even though the row count does not change. `aggregate_country_revenue()` also dumps the full report table into the log, which becomes noisy and non-scalable as data grows. The guideline needs explicit exclusions for value-only and order-only transformations, otherwise the model overlogs low-signal steps.

### 3. New and Updated Guidelines that worked for Problem B_3

#### Guideline 1 (Updated): Log only row-set-changing or shape-changing decision points with counts, and explicitly exclude value-only and order-only transformations

**Description**

When prompting an LLM to add logging in a data pipeline, explicitly restrict logs to stages that change which rows survive or change the table shape in an operationally meaningful way. Require before/after counts, dropped counts, or output-group counts. Explicitly forbid logs for stages that only add labels or columns, reorder rows, or dump full table contents unless that detail is specifically requested.

**Reasoning**

LLMs follow broad instructions too literally. If the prompt only says to log meaningful data changes, the model often treats any transformation as log-worthy, including label-only or sort-only steps. Making the exclusions explicit gives the model a much clearer boundary for what should count as a logging event. This keeps logs concise, improves signal-to-noise ratio, and makes the output more useful for production monitoring.

**Prompt and Context:**  

```
You are given the following Python starter code for a Pandas ETL join-and-report pipeline.

Apply this updated guideline exactly:
Guideline 1 (Updated): Log only row-set-changing or shape-changing decision points with counts, and explicitly exclude value-only and order-only transformations.
When instrumenting a data-processing pipeline:
- Log only stages that change which rows survive or change the table's shape in an operationally meaningful way.
- Include before/after counts, dropped counts, or output-group counts.
- Do NOT log stages that only add labels/columns, reorder rows, or dump full table contents unless explicitly requested.
- Use WARNING when rows are missing or dropped unexpectedly.
- Keep at most one INFO summary log per major stage.

Concrete rules for this file:
- load_inputs(): 1 INFO log with transaction and user counts.
- join_transactions_with_users(): 1 INFO log with before/after row counts; WARNING if joined rows have missing user info.
- label_join_outcomes(): no logging.
- filter_active_users(): 1 INFO log with before/after counts; WARNING if rows were dropped.
- aggregate_country_revenue(): 1 INFO log with number of output countries only; do not log the table.
- sort_report(): no logging.
- build_country_revenue_report(): 1 INFO log with final report row count.

Task:
Add logging that helps a developer understand what happens during these later stages.
Use Python's logging library with a module-level logger named etl_report.
Use plain logging format strings with %d placeholders, not f-strings.
Return the complete modified join_and_report.py file only.

Model:
copilot --model gpt-5.4-mini
```

~~~python
def join_transactions_with_users(
    transactions: pd.DataFrame, users: pd.DataFrame
) -> pd.DataFrame:
    before = len(transactions)
    merged = transactions.merge(users, on="user_id", how="left")
    after = len(merged)
    logger.info(
        "Joined transactions with users: %d rows before, %d rows after.",
        before,
        after,
    )
    missing_users = merged["status"].isna().sum()
    if missing_users > 0:
        logger.warning("%d joined rows are missing user info after join.", missing_users)
    return merged
~~~

~~~python
def filter_active_users(merged: pd.DataFrame) -> pd.DataFrame:
    before = len(merged)
    filtered = merged[merged["status"] == "active"].copy()
    after = len(filtered)
    dropped = before - after
    logger.info(
        "Filtered active users: %d rows before, %d rows after.",
        before,
        after,
    )
    if dropped > 0:
        logger.warning("%d rows were dropped when filtering to active users.", dropped)
    return filtered
~~~

~~~python

The result is shorter, clearer, and much more useful for operators who need to understand where data was lost or transformed.
