### 1.3. Remarks

Under Prompt A (Guideline 4: Decompose by Logical Sub-Behaviors and Test Each Responsibility Separately), Copilot produced well-structured and locally isolated tests. Shipping was validated independently against the raw subtotal threshold and coupon override cases. However, it never generated a threshold-crossing interaction case where discount logic alters shipping eligibility.

The hidden bug depended precisely on such an interaction:

Shipping was computed using the original subtotal, rather than the post-discount subtotal.

Prompt A validated shipping behavior only in isolation. It verified:

- subtotal ≥ 50 → free shipping
- subtotal < 50 → paid shipping

But it did not test scenarios where a discount reduces the subtotal below the threshold after shipping eligibility is evaluated. Therefore, it failed to expose the bug.

In contrast, Prompt B (Guideline 11: Explicitly Test Interaction / Precedence Matrix Across Rules) required the generation of threshold-crossing interaction tests. Copilot generated cases such as:

- subtotal initially ≥ 50,
- VIP discount reduces it below 50,
- shipping should now be charged.

These tests would fail under the buggy implementation because shipping was computed from the wrong base amount.

As a result:

- Interaction between discount and shipping logic was explicitly exercised.
- Threshold-crossing behavior was intentionally tested.
- The implementation bug became observable.

This counterexample shows that Guideline 4 improves structural organization but does not enforce detection of cross-rule threshold interactions. Guideline 11 strengthens test robustness by explicitly requiring interaction-precedence analysis.

Therefore, Guideline 4 is locally helpful but structurally insufficient for detecting threshold-dependent interaction bugs without explicit interaction testing.

### 2.3. Remarks

The old guideline (Guideline 5) was useful for producing many runnable tests, but it did not reliably anchor the generated assertions to the README contract. ### 3.2. New and Updated Guidelines that worked for Problem C

#### Updated Guideline 2: Spec-Anchored Generate–Validate–Repair Loop

**Description of the fix:**

We add a **spec-anchoring classification step** before any repair. Instead of asking the LLM to simply "fix the tests," we require it to compare each failing assertion against the original specification (docstrings, prompt requirements, API contracts) and classify the failure:

- **(A) TEST BUG** — the assertion contradicts the spec → fix the test.
- **(B) IMPL BUG** — the assertion matches the spec but the code disagrees → keep the assertion, mark `@pytest.mark.xfail(reason="IMPL BUG: ...")` so the suite is runnable.
- **(C) UNCLEAR** — the spec is ambiguous → flag for human review.

This preserves bug-detecting tests rather than deleting them.

**Prompt and Context:**

```
Running the tests produced 9 failures. For each failure:

1. Compare the test's expected value against the ORIGINAL SPECIFICATION
   (the InventoryService docstring says "Returns True if available";
   the prompt spec says "bundle discount quantity >= 3"; "SUMMER20 capped
   at $30"; "tax on post-discount amount"; "shipping on discounted subtotal";
   "FLASH5 incompatible with VIP").

2. Classify each failure as:
   (A) TEST BUG  — the test contradicts the spec. Fix the test.
   (B) IMPL BUG  — the test matches the spec but the code disagrees.
       Keep the spec-correct assertion. Mark with @pytest.mark.xfail(
       reason="IMPL BUG: <description>") so the suite runs green.
   (C) UNCLEAR   — spec is ambiguous. Flag for human review.

3. Do not modify checkout_service.py.
4. Return the corrected test file.
```

**Solution or Output generated by GitHub Copilot:**

The LLM produced `test_checkout_v3_spec_anchored.py`. This time, for each failure it checked the spec first:

- **Stock check**: InventoryService docstring says `True` = available. Code raises on `True`. → Classified as **(B) IMPL BUG**. Kept spec-correct assertion with xfail.
- **Bundle boundary**: Spec says `>= 3`. Code uses `> 3`. → **(B) IMPL BUG**. Kept spec assertion, xfailed.
- **SUMMER20**: Spec says "capped at $30" (i.e., `min`). Code uses `max`. → **(B) IMPL BUG**.
- **Tax**: Spec says "post-discount". Code uses subtotal. → **(B) IMPL BUG**.
- **Shipping**: Spec says "discounted subtotal". Code uses subtotal. → **(B) IMPL BUG**.
- **FLASH5+VIP**: Spec says incompatible. Code allows it. → **(B) IMPL BUG**.

No failures were classified as (A) test bugs or (C) unclear — all 6 were correctly identified as implementation bugs.

### 3.3. Remarks

**Why this works:**

The key difference is the **A/B/C classification anchored to the spec**. The original Guideline 2 asks "are these bugs in the tests or the implementation?" without providing a decision framework. The LLM defaults to trusting the code. The updated guideline forces the LLM to check each assertion against the documented contract _before_ deciding what to fix. This creates a bias toward preserving spec-correct assertions rather than destroying them.

The `xfail` mechanism is also important — it means the suite stays green (satisfying the "validate" step of the loop) while preserving the bug-detecting assertions as living documentation.

**Summary of changes to Guideline 2:**

| Original Guideline 2                                 | Updated Guideline 2                                                                    |
| ---------------------------------------------------- | -------------------------------------------------------------------------------------- |
| "Are these bugs in the tests or the implementation?" | "Classify each failure as TEST BUG / IMPL BUG / UNCLEAR by comparing against the spec" |
| Implicitly treats the code as the oracle             | Explicitly treats the spec as the oracle                                               |
| Repairs all failures → green suite                   | Repairs test bugs, xfails impl bugs → green suite + preserved bug signals              |
| No traceability for repair decisions                 | Each xfail has a reason string documenting the spec violation                          |
