## 1. Counterexample Problems

### Problem A_1: Inventory Reorder System

**Task Description:**
The Inventory Reorder System decides when and how much stock to reorder. It performs six sequential steps inside a single `process_reorder` method:

1. **Reorder decision** — compare current stock to the reorder point.
2. **Quantity calculation** — take the larger of deficit-to-max and pipeline demand.
3. **Volume discount lookup** — match quantity against tiered thresholds.
4. **Cost computation** — subtotal, discount amount, total cost.
5. **Safety-stock adjustment** — bump quantity if it falls at or below a safety minimum.
6. **Approval check** — flag orders whose total cost meets or exceeds the auto-approve limit.

The code contains **three bugs**, all of which involve interactions between steps rather than errors within any single step:

| # | Location | Bug | Correct fix |
|---|----------|-----|-------------|
| 1 | Step 5 — safety check | `reorder_qty < safety_minimum` (off-by-one) | Should be `<=` |
| 2 | Step 5 — after quantity bump | Discount, subtotal, discount amount, and total cost are **not recalculated** after the quantity changes | Must recompute cost fields using the new quantity |
| 3 | Step 6 — approval threshold | `total_cost > self.AUTO_APPROVE_LIMIT` (off-by-one) | Should be `>=` |

Bug #2 is the critical one: it is a **pipeline interaction bug** — a later step (safety-stock adjustment) mutates a value computed by earlier steps (quantity), but the downstream fields (discount, costs) that depend on that value are never refreshed. Each step is individually correct in isolation; the error only appears when the steps execute sequentially.

**Starter Code:**

```python
"""
Inventory Reorder System
========================
Determines when and how much stock to reorder based on current inventory
levels, demand forecasts, and supplier discount tiers. Handles:
  - Reorder-point triggering
  - Quantity calculation (deficit vs. pipeline demand)
  - Volume-based discount lookup
  - Cost computation (subtotal, discount, total)
  - Safety-stock adjustment
  - Approval-threshold check
"""


class InventoryReorderSystem:
    DISCOUNT_TIERS = [
        (500, 0.15),   # 15% for 500+ units
        (200, 0.10),   # 10% for 200+ units
        (50,  0.05),   #  5% for  50+ units
    ]
    AUTO_APPROVE_LIMIT = 5000  # orders at or above this need manager sign-off

    def process_reorder(self, current_stock, reorder_point, max_stock,
                         unit_price, daily_usage, lead_time_days):
        """
        Decide whether to reorder and, if so, compute the full order summary.

        Returns a dict with either {"reorder_needed": False} or a full
        breakdown including quantity, discount, costs, and approval flag.
        """
        # Step 1: reorder decision
        if current_stock > reorder_point:
            return {"reorder_needed": False}

        # Step 2: quantity calculation
        deficit = max_stock - current_stock
        pipeline_demand = daily_usage * lead_time_days
        reorder_qty = max(deficit, pipeline_demand)

        # Step 3: volume discount lookup
        discount_pct = self._get_discount(reorder_qty)

        # Step 4: cost computation
        subtotal = reorder_qty * unit_price
        discount_amount = subtotal * discount_pct
        total_cost = subtotal - discount_amount

        # Step 5: safety-stock adjustment
        safety_minimum = int(daily_usage * lead_time_days * 1.5)
        if reorder_qty < safety_minimum:          # BUG: should be <=
            reorder_qty = safety_minimum + 10
            # BUG: discount_pct, subtotal, discount_amount, total_cost
            #      are NOT recalculated after quantity change

        # Step 6: approval check
        needs_approval = total_cost > self.AUTO_APPROVE_LIMIT  # BUG: should be >=

        return {
            "reorder_needed": True,
            "quantity": reorder_qty,
            "discount_pct": discount_pct,
            "subtotal": round(subtotal, 2),
            "discount_amount": round(discount_amount, 2),
            "total_cost": round(total_cost, 2),
            "needs_approval": needs_approval,
        }

    def _get_discount(self, quantity):
        """Return the highest applicable discount percentage."""
        for threshold, discount in self.DISCOUNT_TIERS:
            if quantity >= threshold:
                return discount
        return 0.0
```

### 2. Guidelines that didn't work for Problem A_1

#### Guideline 4: Decompose Complex Methods Before Asking for Tests

Guideline 4 instructs us to break a complex method into its logical sub-behaviors and generate tests for each one independently. For `process_reorder`, the six steps decompose naturally into six test groups. Each group tests its own step with inputs chosen to exercise that step's logic. The problem is that **Bug #2 — stale cost after safety-stock adjustment — spans two steps**. When we test Step 5 in isolation we check that the quantity is bumped correctly, and when we test Step 4 in isolation we check that costs are computed correctly for a given quantity. Neither group verifies that the cost fields are *recalculated* after Step 5 changes the quantity, because that property only exists at the boundary between steps.

### 3. New and Updated Guidelines that worked for Problem A_1

#### Guideline 4: Decompose Complex Methods *and* Generate Pipeline Integration Tests

The original Guideline 4 says:

> *"Break complex methods into sub-behaviors and generate tests for each."*

The updated guideline adds a third step:

> **Step 3 — Pipeline integration tests.** After generating decomposed unit tests for each sub-behavior, generate integration tests that verify *state consistency across sequential steps*. For every variable that is written in one step and read in a later step, add a test that:
>
> 1. Triggers the earlier step's mutation (e.g., safety-stock bumps the quantity).
> 2. Asserts that all downstream fields (discount, subtotal, total cost, approval) are recomputed to reflect the mutated value.
>
> This catches "stale variable" bugs that decomposed tests miss by design.

**Prompt and Context:**

```
I have a Python class InventoryReorderSystem with a method process_reorder
that performs six sequential steps:
  1. Reorder decision (compare current_stock to reorder_point)
  2. Quantity calculation (max of deficit and pipeline demand)
  3. Volume discount lookup (tiered thresholds)
  4. Cost computation (subtotal, discount amount, total cost)
  5. Safety-stock adjustment (bump quantity if below safety minimum)
  6. Approval check (flag orders above auto-approve limit)

Please:
(A) Decompose these into six independent test groups using pytest, covering
    typical values, edge cases, and boundary conditions for each step.

(B) ADDITIONALLY, generate pipeline integration tests that verify state
    consistency across sequential steps. Specifically:
    - When Step 5 changes the quantity, verify that discount_pct, subtotal,
      discount_amount, and total_cost reflect the NEW quantity, not the old one.
    - When the total cost changes due to recalculation, verify that
      needs_approval reflects the NEW total cost.
    Choose input values where the safety-stock adjustment triggers AND
    the recalculated cost crosses the approval threshold.
```

## 1. Counterexample Problems

### Problem A_2: Generate–Validate–Repair Loop Degeneration

**Task Description:**

The Generate–Validate–Repair loop (Guideline 2) suggests that instead of one-shot generation, LLM-generated tests should go through an iterative refinement cycle:

1. Generate tests  
2. Run tests  
3. Parse failures  
4. Feed failures back  
5. Regenerate  
6. Re-validate  

The assumption is that this loop improves correctness over time.

However, this counterexample demonstrates a failure mode where the repair loop degenerates into assertion weakening and false-positive convergence.

---

**Starter Code:**

```python
def divide(a: float, b: float) -> float:
    if b == 0:
        raise ValueError("division by zero")
    return a / b
```

The intended contract:

- Raises ValueError when b == 0  
- Returns a / b otherwise

If the user continues providing vague repair prompts such as:

```
Tests still failing. Fix.
```

The model may:

- Replace assertions with `assert True`
- Wrap all tests in broad try/except
- Modify expected outputs instead of diagnosing implementation errors
- Suggest altering the implementation

This produces a pathological cycle:

Generate → Fail → Over-general repair → False pass → Silent quality loss

---

### Root Cause Analysis

The generate–validate–repair loop assumes:

Failure feedback + LLM reasoning → improved alignment

But without guardrails:

- The model optimizes for minimizing surface errors.
- It weakens assertions.
- It removes discriminative power.
- It overfits to error messages.

The loop becomes an error amplification mechanism rather than a correction mechanism.

---

### 3. Updated Guideline that worked for Problem A_2

#### Updated Guideline 2: Repair with Contract Preservation Constraints

When entering the repair phase:

1. Restate the intended contract explicitly.
2. Require explanation of root cause before modification.
3. Forbid weakening assertions.
4. Forbid broad exception catching.
5. Require strong assertions using pytest.raises where applicable.

---

**Revised Repair Prompt:**

```
The test failed with:
ValueError: division by zero

Contract:
- divide(10, 0) must raise ValueError.

Explain whether the failure indicates a test bug or implementation bug.
Do not weaken assertions.
Return corrected pytest tests.
```

---

**Corrected Output:**

```python
import pytest

def test_divide_zero():
    with pytest.raises(ValueError):
        divide(10, 0)
```

---

### Results After Applying Updated Guideline

- The test preserves the contract.
- It verifies the correct exception type.
- It fails if the implementation changes behavior.
- It does not degrade assertion strength.

The loop now converges toward correctness instead of pass-optimization.

### New Guideline: For End to End (E2E) testing, specify algorithms and step-by-step behavior for non-trivial test cases

**Description:**
When working with complicated multi-step E2E testing, provide all information on what logic the code uses to decide what happens next, so the test can re-create what should happen, and then compare what does happen to what should happen. If this includes several different steps, include very clear details for what should be done at each step, and what should be asserted at every step.

**Reasoning:**
E2E testing can be much more involved than usual code testing due to api-calls, and interacting with front-end. When it does not fully reconstruct an algorithm, it can have incorrect assertions at later steps. It can also focus on only parts of the evaluation, so explicitly stating what must be checked at every single step ensures that it looks at all values to assert that values are correct.

describe('Sorting App - initial screen', () => {
  it('shows Choose File, Load Demo, Continue Session', () => {
    cy.visit('http://localhost:3000');
    cy.contains('Choose File').should('be.visible');
    cy.contains('Load Demo').should('be.visible');
    cy.contains('Continue Session').should('be.visible');
  });

  it('loads demo and shows Ranking Settings', () => {
    cy.visit('http://localhost:3000');
    cy.contains('Load Demo').click();
    cy.contains('Ranking Settings', { timeout: 10000 }).should('be.visible');
  });

  it('validates binary-insertion comparisons and final sorted order', () => {
    // demo items (from FileSelection.tsx)
    const items = [
      { Name: 'Item 1', Rating: '8.5' },
      { Name: 'Item 2', Rating: '7.2' },
      { Name: 'Item 3', Rating: '9.1' },
      { Name: 'Item 4', Rating: '6.8' },
      { Name: 'Item 5', Rating: '8.9' },
      { Name: 'Item 6', Rating: '7.5' },
      { Name: 'Item 7', Rating: '8.2' },
      { Name: 'Item 8', Rating: '6.9' },
      { Name: 'Item 9', Rating: '9.3' },
      { Name: 'Item 10', Rating: '7.8' }
    ]

    // Simulate server-side binary-insertion sequence to get expected comparisons and final order
    const unsorted = Array.from({ length: items.length }, (_, i) => i)
    const sorted = []
    const comparisons = []

    while (unsorted.length > 0) {
      const current = unsorted.shift()
      let low = 0
      let high = sorted.length

      if (low >= high) {
        sorted.splice(low, 0, current)
        continue
      }

      while (low < high) {
        const mid = Math.floor((low + high) / 2)
        comparisons.push({ left: current, right: sorted[mid] })

        const currRating = parseFloat(items[current].Rating)
        const rightRating = parseFloat(items[sorted[mid]].Rating)

        if (currRating >= rightRating) {
          high = mid
        } else {
          low = mid + 1
        }
      }

      sorted.splice(low, 0, current)
    }

    const expectedFinal = sorted.map(i => items[i].Name)

    // Start the app and run through comparisons, validating left/right items each time
    cy.visit('http://localhost:3000')
    cy.contains('Load Demo').click()
    cy.contains('Start Ranking →', { timeout: 10000 }).click()

    // Step through expected comparisons in order
    comparisons.forEach((cmp) => {
      // Wait for comparison panels to render
      cy.get('.comparison-panel.left .item-field-label', { timeout: 10000 }).should('exist')

      // Read left Name and Rating
      cy.get('.comparison-panel.left .item-field-label').then(($leftFields) => {
        const leftArr = Array.from($leftFields).map(e => e.innerText || '')
        const leftNameLine = leftArr.find(t => t.includes('Name')) || ''
        const leftRatingLine = leftArr.find(t => t.includes('Rating')) || ''
        const leftName = leftNameLine.split(':').slice(1).join(':').trim()
        const leftRating = parseFloat((leftRatingLine.match(/([0-9]+\.?[0-9]*)/) || [])[1])

        // Read right Name and Rating and then assert and click accordingly
        cy.get('.comparison-panel.right .item-field-label').then(($rightFields) => {
          const rightArr = Array.from($rightFields).map(e => e.innerText || '')
          const rightNameLine = rightArr.find(t => t.includes('Name')) || ''
          const rightRatingLine = rightArr.find(t => t.includes('Rating')) || ''
          const rightName = rightNameLine.split(':').slice(1).join(':').trim()
          const rightRating = parseFloat((rightRatingLine.match(/([0-9]+\.?[0-9]*)/) || [])[1])

          // Verify panels show the expected items for this comparison
          expect(leftName).to.equal(items[cmp.left].Name)
          expect(rightName).to.equal(items[cmp.right].Name)

          // Click the correct button (choose the higher rating)
          if (leftRating >= rightRating) {
            cy.contains('✓ PREFER LEFT').click()
          } else {
            cy.contains('PREFER RIGHT ✓').click()
          }
        })
      })
    })

    // After all comparisons, verify results screen and final order
    cy.contains('Your Ranked Results', { timeout: 120000 }).should('be.visible')
    cy.get('.itemlist').within(() => {
      cy.get('h4').then(($els) => {
        const texts = Array.from($els).map(e => (e.textContent || '').trim())
        expect(texts).to.deep.equal(expectedFinal)
      })
    })
  })
});
