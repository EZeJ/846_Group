### 1.3. Remarks

Under Prompt A (Guideline 4: Decompose by Logical Sub-Behaviors and Test Each Responsibility Separately), Copilot produced well-structured and locally isolated tests. Shipping was validated independently against the raw subtotal threshold and coupon override cases. However, it never generated a threshold-crossing interaction case where discount logic alters shipping eligibility.

The hidden bug depended precisely on such an interaction:

Shipping was computed using the original subtotal, rather than the post-discount subtotal.

Prompt A validated shipping behavior only in isolation. It verified:

- subtotal ≥ 50 → free shipping
- subtotal < 50 → paid shipping

But it did not test scenarios where a discount reduces the subtotal below the threshold after shipping eligibility is evaluated. Therefore, it failed to expose the bug.

In contrast, Prompt B (Guideline 11: Explicitly Test Interaction / Precedence Matrix Across Rules) required the generation of threshold-crossing interaction tests. Copilot generated cases such as:

- subtotal initially ≥ 50,
- VIP discount reduces it below 50,
- shipping should now be charged.

These tests would fail under the buggy implementation because shipping was computed from the wrong base amount.

As a result:

- Interaction between discount and shipping logic was explicitly exercised.
- Threshold-crossing behavior was intentionally tested.
- The implementation bug became observable.

This counterexample shows that Guideline 4 improves structural organization but does not enforce detection of cross-rule threshold interactions. Guideline 11 strengthens test robustness by explicitly requiring interaction-precedence analysis.

Therefore, Guideline 4 is locally helpful but structurally insufficient for detecting threshold-dependent interaction bugs without explicit interaction testing.

**Why this is bad:**

1. **Decomposition is forced on single-responsibility code.**
`clamp()` has one responsibility: return value clamped to [lo, hi]. There are no separate “validate → transform → persist” steps. Guideline 4 asks the model to list distinct behaviors and test each one, so it invents behaviors like “below bound,” “above bound,” “in range,” and “validation.” Those are input cases for the same behavior, not separate responsibilities, which leads to redundant test groups and noisy structure.

2. **The model is steered toward invented structure instead of real coverage.**
The guideline pushes the model to create a test group per “behavior.” The result is many small test methods that could be a few parametrized tests. The model prioritizes structure (multiple classes and methods) over clarity and over targeting real faults (e.g. off-by-one at boundaries). So the test suite looks organized but is shallow and may not catch simple bugs like <= vs <.

3. **“Negative cases per behavior” invents a contract the code doesn’t have.**
The spec says the function assumes lo <= hi and does not validate. Following “boundary + negative per behavior,” the model adds tests for None and for lo > hi. That invents a validation responsibility the code doesn’t have, so tests are either underspecified (e.g. for lo > hi) or irrelevant (e.g. for None), and they don’t reflect the documented contract.

The model may never produce a compact, fault-focused suite, may add tests for non-existent validation, and may miss boundary bugs. So Guideline 4, as applied here, does not work as intended for this question.

#### Guideline 2 (for guideline 2): Match Test Structure to Code Complexity

**Description:**  

**Do not force decomposition on every method.** First decide whether the method under test has one clear responsibility or several distinct sub-behaviors.

- **Single-responsibility or trivial code** (e.g. pure functions, thin wrappers, one clear “do X”):  
  Ask for **one focused test suite** that covers:
  - Boundary and negative cases (as in Guideline 3 (in-class)),
  - A small **fault model** (as in Guideline 5 (in-class)) if the code has plausible near-correct bugs,
  - Strong, exact assertions.
  Do **not** ask the LLM to “list distinct behaviors” or “generate tests per behavior” for such code.

- **Multi-responsibility code** (e.g. validate → transform → persist → log):  
  Then apply **Guideline 4**: list documentable sub-behaviors, generate separate test groups per sub-behavior, and keep boundary/negative/fault-model within each group.

**Rule of thumb:** If you cannot write 2–3 distinct, non-overlapping sentence descriptions of what the method does (e.g. “validates input”, “computes price”, “saves to DB”), treat it as single-responsibility and skip decomposition.

**Reasoning:** 

1. **Guideline 4’s rationale** (Wang et al. [4]) is that test quality degrades on **long, multi-responsibility** methods because the model loses track of which behavior it is testing. Decomposition helps there. For **short, single-responsibility** methods there is no such confusion—there is only one behavior. Forcing decomposition has no benefit and causes the LLM to invent fake “behaviors” (e.g. “handles below bound”, “handles above bound”) that are just input categories, leading to redundant structure and sometimes invented contracts (e.g. “validation”).

2. **Single-responsibility code** is better tested by:
   - A small set of **boundary/negative** cases, and/or
   - A **fault model** with one test per plausible bug,
   rather than by artificial “behavior” groups.

3. **Consistency and auditability:** When the test structure matches the real structure of the code (one behavior → one suite; many behaviors → one suite per behavior), it is easier to audit coverage and to know what is being tested. Invented behaviors make the test suite misleading.

```
   - value in range (unchanged)
   - value == lo and value == hi (boundary)

   - FM3: Returns value when value < lo (forgets to clamp low)

```

### Problem A_1: Inventory Reorder System

**Task Description:**
The Inventory Reorder System decides when and how much stock to reorder. It performs six sequential steps inside a single `process_reorder` method:

1. **Reorder decision** — compare current stock to the reorder point.
2. **Quantity calculation** — take the larger of deficit-to-max and pipeline demand.
3. **Volume discount lookup** — match quantity against tiered thresholds.
4. **Cost computation** — subtotal, discount amount, total cost.
5. **Safety-stock adjustment** — bump quantity if it falls at or below a safety minimum.
6. **Approval check** — flag orders whose total cost meets or exceeds the auto-approve limit.

The code contains **three bugs**, all of which involve interactions between steps rather than errors within any single step:

| # | Location | Bug | Correct fix |
|---|----------|-----|-------------|
| 1 | Step 5 — safety check | `reorder_qty < safety_minimum` (off-by-one) | Should be `<=` |
| 2 | Step 5 — after quantity bump | Discount, subtotal, discount amount, and total cost are **not recalculated** after the quantity changes | Must recompute cost fields using the new quantity |
| 3 | Step 6 — approval threshold | `total_cost > self.AUTO_APPROVE_LIMIT` (off-by-one) | Should be `>=` |

Bug #2 is the critical one: it is a **pipeline interaction bug** — a later step (safety-stock adjustment) mutates a value computed by earlier steps (quantity), but the downstream fields (discount, costs) that depend on that value are never refreshed. Each step is individually correct in isolation; the error only appears when the steps execute sequentially.

**Starter Code:**

```python
"""
Inventory Reorder System
========================
Determines when and how much stock to reorder based on current inventory
levels, demand forecasts, and supplier discount tiers. Handles:
  - Reorder-point triggering
  - Quantity calculation (deficit vs. pipeline demand)
  - Volume-based discount lookup
  - Cost computation (subtotal, discount, total)
  - Safety-stock adjustment
  - Approval-threshold check
"""


class InventoryReorderSystem:
    DISCOUNT_TIERS = [
        (500, 0.15),   # 15% for 500+ units
        (200, 0.10),   # 10% for 200+ units
        (50,  0.05),   #  5% for  50+ units
    ]
    AUTO_APPROVE_LIMIT = 5000  # orders at or above this need manager sign-off

    def process_reorder(self, current_stock, reorder_point, max_stock,
                         unit_price, daily_usage, lead_time_days):
        """
        Decide whether to reorder and, if so, compute the full order summary.

        Returns a dict with either {"reorder_needed": False} or a full
        breakdown including quantity, discount, costs, and approval flag.
        """
        # Step 1: reorder decision
        if current_stock > reorder_point:
            return {"reorder_needed": False}

        # Step 2: quantity calculation
        deficit = max_stock - current_stock
        pipeline_demand = daily_usage * lead_time_days
        reorder_qty = max(deficit, pipeline_demand)

        # Step 3: volume discount lookup
        discount_pct = self._get_discount(reorder_qty)

        # Step 4: cost computation
        subtotal = reorder_qty * unit_price
        discount_amount = subtotal * discount_pct
        total_cost = subtotal - discount_amount

        # Step 5: safety-stock adjustment
        safety_minimum = int(daily_usage * lead_time_days * 1.5)
        if reorder_qty < safety_minimum:          # BUG: should be <=
            reorder_qty = safety_minimum + 10
            # BUG: discount_pct, subtotal, discount_amount, total_cost
            #      are NOT recalculated after quantity change

        # Step 6: approval check
        needs_approval = total_cost > self.AUTO_APPROVE_LIMIT  # BUG: should be >=

        return {
            "reorder_needed": True,
            "quantity": reorder_qty,
            "discount_pct": discount_pct,
            "subtotal": round(subtotal, 2),
            "discount_amount": round(discount_amount, 2),
            "total_cost": round(total_cost, 2),
            "needs_approval": needs_approval,
        }

    def _get_discount(self, quantity):
        """Return the highest applicable discount percentage."""
        for threshold, discount in self.DISCOUNT_TIERS:
            if quantity >= threshold:
                return discount
        return 0.0
```

### 3. New and Updated Guidelines that worked for Problem A_1

#### Guideline 4: Decompose Complex Methods *and* Generate Pipeline Integration Tests

The original Guideline 4 says:

> *"Break complex methods into sub-behaviors and generate tests for each."*

The updated guideline adds a third step:

> **Step 3 — Pipeline integration tests.** After generating decomposed unit tests for each sub-behavior, generate integration tests that verify *state consistency across sequential steps*. For every variable that is written in one step and read in a later step, add a test that:
>
> 1. Triggers the earlier step's mutation (e.g., safety-stock bumps the quantity).
> 2. Asserts that all downstream fields (discount, subtotal, total cost, approval) are recomputed to reflect the mutated value.
>
> This catches "stale variable" bugs that decomposed tests miss by design.
